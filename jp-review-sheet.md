プロジェクト Inception 評価スケール（Scale for project Inception）
このチームの学生1名を評価してください。

はじめに（Introduction）
以下のルールを厳守してください：

評価プロセス全体を通じて、常に礼儀正しく、品位を保ち、敬意を払い、建設的な態度を維持してください。コミュニティの健全性はこれにかかっています。

評価対象である学生またはグループのプロジェクトにおける、潜在的な不具合（機能不全）を特定してください。特定された問題について、時間をかけて議論やディベートを行ってください。

ピア（仲間）によって、プロジェクトの指示や機能の範囲に対する理解に多少の差異がある可能性を考慮しなければなりません。常にオープンな心を持ち、可能な限り誠実に採点してください。ピア評価が真剣に行われて初めて、教育的効果が生まれます。

ガイドライン（Guidelines）
評価対象の学生またはグループのGitリポジトリに提出された成果物のみを採点してください。

Gitリポジトリが本当にその学生（または学生たち）のものであるか、ダブルチェック（再確認）してください。プロジェクトが指定されたものであることを確認してください。また、git clone が空のディレクトリで行われていることを確認してください。

公式リポジトリの内容ではないものを評価させようとする、あなたを欺くための悪意のあるエイリアス（別名）が使用されていないか慎重に確認してください。

不意の事態を防ぐため、また該当する場合は、採点を円滑にするために使用されるスクリプト（テストスクリプトや自動化スクリプトなど）を一緒にレビューしてください。

自分がまだ評価対象の課題を完了していない場合は、評価プロセスを開始する前にサブジェクト（課題文）の全体を読まなければなりません。

空のリポジトリ、機能しないプログラム、Norm（規約）エラー、不正行為（カンニングなど）を報告するために、用意されているフラグを使用してください。
これらのケースでは評価プロセスがその時点で終了し、最終成績は 0点（不正行為の場合は -42点）になります。ただし、不正行為の場合を除き、将来繰り返すべきでない間違いを特定するために、提出された成果物を一緒にレビューすることを強く推奨します。

Dockerの基本（Docker Basics）
まず Dockerfile の確認から始めてください。サービスごとに1つの Dockerfile がなければなりません。Dockerfile が空のファイルではないことを確認してください。そうでない場合、または Dockerfile が見つからない場合、評価プロセスはここで終了します。

評価対象の学習者が自身で Dockerfile を記述し、独自の Docker イメージをビルドしたことを確認してください。既製品（公式の既成イメージ等）を使用することや、DockerHub などのサービス（そのままのサービスイメージ）を使用することは禁止されています。

すべてのコンテナが Alpine/Debian の最後から2番目の安定版からビルドされていることを確認してください。Dockerfile が FROM alpine:X.X.X や FROM debian:XXXXX、またはその他のローカルイメージで始まっていない場合、評価プロセスはここで終了します。

Docker イメージは、対応するサービスと同じ名前でなければなりません。そうでない場合、評価プロセスはここで終了します。

Makefile が docker compose を介してすべてのサービスをセットアップしていることを確認してください。つまり、コンテナが docker compose を使用してビルドされ、クラッシュが発生していない必要があります。そうでない場合、評価プロセスは終了します。

構成の変更（Configuration modification）

サービスは、新しい構成でもアクセス可能であり、機能し続けなければなりません。

ボーナス（Bonus）
このプロジェクトのボーナスパートは、シンプルであることを意図しています。追加のサービスごとに Dockerfile を記述しなければなりません。したがって、各サービスはそれぞれ独自のコンテナ内で実行され、必要に応じて専用のボリュームを持ちます。ボーナスパートは、必須パートが完全に、かつ完璧に完了している場合にのみ評価してください。「完璧」とは、必須パートがすべて完了しており、不具合なく動作していることを意味します。必須要件を「すべて（ALL）」クリアしていない場合、ボーナスパートは一切評価されません。
